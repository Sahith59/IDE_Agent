The IDE Expert Agent is not a simple code search tool or a documentation chatbot. It is a multi-dimensional expert system that holds knowledge across every layer of your product simultaneously:

Technical knowledge: every file, class, function, and line of code
Business knowledge: rules, workflows, edge cases, and product logic
Architectural knowledge: why decisions were made, how components connect
Operational knowledge: known bugs, workarounds, deployment considerations
Tribal knowledge: insights that exist only in your head — which you feed to it over time

4. Knowledge Base Design

The quality of the agent's answers is directly proportional to the quality and richness of its knowledge base. This section defines what goes in, how it is structured, and how to continuously expand it.

4.1 Knowledge Sources — What to Feed the Agent
Source 1: The Git Repository (Automated)
Everything in your IDE repo is automatically ingested:
All source code files — parsed with language-aware chunking that respects function and class boundaries
Configuration files — package.json, tsconfig, pyproject.toml, docker-compose, etc.
README and all markdown files — product descriptions, setup guides, feature docs
Git commit history — last 500+ commits, capturing decisions, bug fixes, and refactors over time
Any inline comments and docstrings — these are often the richest source of intent

Source 2: Your Personal Notes (Most Valuable)

5. Agent Design — The Router & Reasoning System

The agent is not a simple question-and-answer pipeline. It uses a LangGraph-based routing system that classifies each question, selects the right knowledge sources, and constructs a context-rich prompt before generating an answer.

5.1 Question Classification
Every incoming question is first classified into one of five categories. This classification determines which knowledge collections are searched and how the answer is framed:


5.2 The Senior Expert Persona
The LLM is given a carefully designed system prompt that shapes it into a confident, knowledgeable product expert. Key elements of this persona:

Confident and direct — it does not hedge unnecessarily. It knows this product.
Context-aware — it connects code behavior to business intent naturally
Reference-specific — it points to actual file paths, function names, and commit messages
Honest about gaps — if the retrieved context does not cover the question, it says so clearly rather than hallucinating
Multi-audience — it adjusts explanation depth based on whether the question is technical or business-oriented

5.3 Retrieval Strategy — MMR (Maximal Marginal Relevance)
Standard similarity search often returns redundant chunks — five slightly different versions of the same function. The agent uses Maximal Marginal Relevance (MMR) retrieval, which balances relevance with diversity:
Fetches a large candidate pool (top 20 most similar chunks)
Then selects the top 8 that are both relevant AND diverse from each other
Result: the LLM receives a richer, more complete picture of the topic

5.4 Context Window Management
Large models support large context windows (up to 128K tokens in some cases). The agent is configured to use a large context window (8192-32768 tokens) so that multiple relevant code chunks, documentation sections, and commit messages can all be included in a single prompt. This is critical for producing answers that synthesize across multiple sources.

6. Continuous Learning & Knowledge Expansion

One of the most important design goals is the ability to constantly train and improve the agent as you learn more. This section defines the pathways for feeding new knowledge into the system.

6.1 The Three Learning Pathways
Pathway 1: Notes-Based Learning (Daily / Ongoing)
This is the primary learning pathway and requires zero technical effort. You simply write markdown files:

Learn something new about the product (business rule, edge case, architecture insight)
Write it in plain English in the appropriate notes file (e.g., my_notes/billing_rules.md)
Run the re-index command (takes 2-5 minutes)
The agent now knows what you know


Pathway 2: Code Repository Sync (When Repo Changes)
Whenever new features are built, code is refactored, or significant changes are merged into the repo, re-run the full ingestion pipeline. This keeps the agent's understanding of the codebase current:

Set up a simple script that you run after each significant PR merge
The pipeline is incremental-friendly — only changed files need re-embedding (future optimization)
Git history is re-ingested automatically — new commits bring new context

Pathway 3: Fine-Tuning on Q&A Pairs (Quarterly / When Needed)
Over time, as you use the agent and notice gaps or incorrect answers, you can build a dataset of correct Q&A pairs and run lightweight fine-tuning using Unsloth with QLoRA adapters. This teaches the model to answer in your preferred style and with your domain's specific vocabulary:

Collect 50-200 high-quality question + ideal-answer pairs about the IDE product
Format as JSONL with instruction / output fields
Fine-tune the base model using Unsloth (works on laptop GPUs with 4-bit quantization)
The fine-tuned model is converted to GGUF and loaded back into Ollama
Result: a model that has internalized domain knowledge, not just retrieves it

7. Sharing Knowledge With Others

A core goal is for this agent to serve not just you but anyone who needs to understand the IDE product. This section covers how to safely share the agent's capabilities while keeping all data local.

7.1 Sharing Modes
Mode 1: Local Network Access (Recommended)
Run the agent on your machine and expose it on your local network. Anyone on the same network (teammates, colleagues at the same office or VPN) can access the Streamlit UI from their browser:

The LLM and vector database never leave your machine
Other users access the UI via your machine's local IP address
No data is transmitted over the internet
Control access by running on a VPN-only network interface

Mode 2: Exported Knowledge Base (Portable)
Export the ChromaDB vector store directory and the notes folder. Anyone with the same setup (Ollama + Python stack) can run their own instance of the agent pointing at the same knowledge base:
Share the vectordb/ folder and my_notes/ folder
Recipient runs the UI locally — the knowledge base is their copy
Good for distributing to remote team members

Mode 3: Knowledge Document Exports
For stakeholders who don't need the agent but need the knowledge, export structured summaries from the notes collection as PDF or Word documents. These can be shared as traditional documentation.

8. Implementation Plan

The following phased plan breaks the build into manageable milestones, ordered by impact and dependency.

Phase 1: Foundation (Day 1-2)
Goal: Get the basic agent running end-to-end with code search.

Install Ollama and pull the chosen LLM (start with qwen2.5-coder:14b if hardware is limited)
Pull the nomic-embed-text embedding model
Install Python dependencies: langchain, langchain-ollama, langchain-community, langgraph, chromadb, streamlit
Clone your IDE repo locally
Run the ingestion script on the code collection only
Test the basic RAG chain with a few code questions

Phase 2: Full Knowledge Base (Day 3-5)
Goal: Ingest all knowledge layers and validate retrieval quality.

Add git history ingestion
Add documentation / README ingestion
Create the my_notes/ folder and write initial business rules and architecture notes
Re-run ingestion across all four collections
Test with business logic questions and architecture questions
Tune chunk sizes and retrieval k values based on answer quality

Phase 3: Intelligent Router (Day 6-8)
Goal: Add the LangGraph routing layer for smart question classification.

Implement the question classifier node
Implement the multi-collection retriever with routing logic
Connect all nodes in the LangGraph state graph
Test all five question types (code, business, debug, architecture, general)
Tune the system prompt persona for the right tone and confidence level

Phase 4: UI & Sharing (Day 9-10)
Goal: Build a clean interface and make it accessible to others.

Build the Streamlit chat UI with conversation history
Add source attribution — show which files/commits informed each answer
Add a notes editor panel — add notes directly from the UI
Expose on local network for team access
Write a README for the agent itself so others can set it up

Phase 5: Upgrade to Large Model (After Validation)
Goal: Upgrade to the largest model your hardware supports for maximum answer quality.

Once the system is validated end-to-end on the smaller model, pull qwen2.5-coder:32b or llama3.3:70b
Test the same benchmark questions — compare answer depth and accuracy
Tune context window size (num_ctx) based on the larger model's capacity
Run re-indexing with mxbai-embed-large for better semantic retrieval if needed

10. Quality Assurance & Testing

Answer quality should be tested systematically, not just informally. This section provides a framework for evaluating and improving the agent.

10.1 Benchmark Question Set
Build a set of 20-30 'golden questions' that you know the correct answers to. Test these after every major change:

5 code questions (e.g., 'What does the parseConfig() function do and where is it?')
5 business logic questions (e.g., 'What happens when a free-tier user tries to create a 4th project?')
5 architecture questions (e.g., 'How does the frontend communicate with the backend API?')
5 debugging questions (e.g., 'Why might a user see a 401 error after their session expires?')
5 general questions (e.g., 'Give me an overview of what the IDE application does')

10.3 When Answers Are Wrong
When the agent gives an incorrect or incomplete answer, do not just accept it. Feed the correction back into the system:

Identify the gap — is it missing knowledge, wrong retrieval, or poor reasoning?
If missing knowledge: write a note in my_notes/ with the correct information
If wrong retrieval: improve chunking or add more specific context to notes
If poor reasoning: consider upgrading to a larger model or improving the system prompt
Re-index and test again
